## Step 3: Detect Logic Dependencies

Scan all tasks for **shared interfaces** -- cases where two tasks by different devs modify different files but share a logical boundary:

1. Look for tasks that produce/consume the same function, API endpoint, data shape, or validation rule.
2. Mark these tasks with `[DEP: Task X.Y]` and note the shared interface:
   ```
   - Shared interface: [DEP: Task 2.1] -- consumes getUserById() return type
   ```
3. For each `[DEP]` pair, recommend:
   - **Merge order**: lower-layer task merges first
   - **Boundary integration test**: create a task to test the shared interface
4. If > 2 tasks share the same boundary, flag it as a risk and recommend a brief alignment meeting.

---

## Step 4: Validate Task Quality

Before presenting to user, verify:
1. Every acceptance criterion in `acceptance.md` maps to at least one task.
2. Every task has a clear definition of done (test + file + description).
3. Dependency chains have no cycles.
4. Parallel tasks `[P]` are truly independent (no shared file edits **and** no shared interfaces).
5. Logic dependencies `[DEP]` are explicitly marked with merge order noted.
6. Effort estimates are realistic (no single task > 8 hours; break down if larger).

---

## Step 5: Present & Confirm

1. Present the refined `tasks.md` to the user.
2. Use the platform confirmation mechanism defined in `rules/09-platform-confirmation.md` to get user approval before proceeding.
3. Once approved: "Tasks ready. Run `/aidlc-build <change-id>` to start TDD implementation."

---

## Rules

- Test tasks come BEFORE implementation tasks within each phase (TDD structure).
- Every task must reference the files it will create or modify.
- For full-spec: respect sprint scope. Do not create tasks outside the sprint commitment.
- `[P]` marker means the task can run in parallel with the preceding task(s).
- `[DEP: Task X.Y]` marker means this task shares a logical interface with another task. Both devs must agree on the interface contract before coding. See `docs/team-conflict-playbook.md` Scenario 6.
- Task IDs must be stable -- do not renumber after assignment.
